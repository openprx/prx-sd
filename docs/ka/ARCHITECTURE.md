# არქიტექტურა

PRX-SD სტრუქტურირებულია როგორც Cargo სამუშაო სივრცე მოდულური crate-ებით, რომელთაგან თითოეული პასუხისმგებელია კონკრეტულ დომენზე.

## სამუშაო სივრცის განლაგება

```
prx-sd/
├── crates/
│   ├── cli/           # "sd" ბინარი — ბრძანების ხაზის ინტერფეისი
│   ├── core/          # სკანირების ძრავის კოორდინაცია
│   ├── signatures/    # ჰეშ მონაცემთა ბაზა (LMDB) + YARA-X წესების ძრავი
│   ├── parsers/       # ბინარული ფორმატების ანალიზატორები
│   ├── heuristic/     # ევრისტიკული შეფასება + ML გამოყვანა
│   ├── realtime/      # ფაილური სისტემის მონიტორინგი + ქსელური ფილტრაცია
│   ├── quarantine/    # დაშიფრული კარანტინის საცავი
│   ├── remediation/   # საფრთხეებზე რეაგირების მოქმედებები
│   ├── sandbox/       # პროცესის იზოლაცია + ქცევის ანალიზი
│   ├── plugins/       # WebAssembly მოდულების გარემო
│   └── updater/       # ხელმოწერების განახლების კლიენტი
├── update-server/     # ხელმოწერების განაწილების სერვერი (Axum)
├── gui/               # დესკტოპ GUI (Tauri 2 + Vue 3)
├── drivers/           # ოპერაციული სისტემის ბირთვის დრაივერები
│   └── windows-minifilter/  # Windows ფაილური სისტემის minifilter (C)
├── signatures-db/     # ჩაშენებული მინიმალური ხელმოწერები
├── packaging/         # განაწილების შეფუთვა
├── tests/             # ინტეგრაციული ტესტები
├── tools/             # აგებისა და უტილიტების სკრიპტები
├── install.sh         # დაყენების სკრიპტი
└── uninstall.sh       # წაშლის სკრიპტი
```

სკანირების ძრავი (`core`) კოორდინაციას უწევს მრავალშრიან აღმოჩენის კონვეიერს:

                    └──────┬───────┘

### შრეების დეტალები

1. **ჰეშ შედარება** -- O(1) ძებნა LMDB მონაცემთა ბაზაში, რომელიც შეიცავს SHA-256 და MD5 ჰეშებს ClamAV-დან, abuse.ch-დან, VirusShare-დან და მორგებული ბლოკ-სიებიდან.

2. **YARA-X წესები** -- შაბლონების შედარება YARA-X ძრავის გამოყენებით (YARA-ს Rust-ნატიური იმპლემენტაცია). წესები იტვირთება ჩაშენებული ნაგულისხმევი წესებიდან და გარე ხელმოწერების საცავიდან.

3. **ევრისტიკული ანალიზი** -- ფაილის ტიპზე დაფუძნებული ანალიზი:
   - **PE:** სექციების ენტროპია, საეჭვო API იმპორტები (CreateRemoteThread, VirtualAllocEx), შეფუთვის აღმოჩენა (UPX, Themida), დროის ანაბეჭდის ანომალიები
   - **ELF:** სექციების ენტროპია, LD_PRELOAD მითითებები, cron/systemd მუდმივობა, SSH backdoor შაბლონები
   - **MachO:** სექციების ენტროპია, dylib ინჟექცია, LaunchAgent მუდმივობა, Keychain წვდომა

4. **ML გამოყვანა** (არასავალდებულო, ფუნქციის დროშა `onnx`) -- ONNX მოდელის შეფასება tract-ის მეშვეობით:
   - PE: 64-განზომილებიანი ნიშნების ვექტორი (იმპორტის ცხრილის ჰეშები, სექციების ენტროპია, API ხელმოწერები)
   - ELF: 48-განზომილებიანი ნიშნების ვექტორი (სექციების ენტროპია, სიმბოლოების ცხრილი, დინამიკური ბიბლიოთეკები)

5. **VirusTotal ღრუბელი** -- სარეზერვო ვარიანტი ლოკალურად ვერნაპოვნი ფაილებისთვის. მოთხოვნის გაგზავნა VirusTotal API-ზე (უფასო დონე: 500 მოთხოვნა/დღეში). შედეგები ქეშირდება LMDB-ში.

### ქულების სისტემა

- ქულა >= 60: **მავნე (Malicious)**
- ქულა 30-59: **საეჭვო (Suspicious)**
- ქულა < 30: **სუფთა (Clean)**

საბოლოო ვერდიქტი არის აღმოჩენის ნებისმიერი შრიდან მიღებული უმაღლესი საფრთხის დონე.

`realtime` crate უზრუნველყოფს უწყვეტ დაცვას მრავალი ქვესისტემის მეშვეობით:

| ქცევის მონიტორინგი | /proc + audit (execve/connect/open) | - | - |

## კარანტინის საცავი

ფაილები კარანტინში მოთავსდება AES-256-GCM ავთენტიფიცირებული დაშიფვრის გამოყენებით:

1. შემთხვევითი 256-ბიტიანი გასაღებისა და 96-ბიტიანი nonce-ის გენერაცია
2. ფაილის შიგთავსის დაშიფვრა AES-256-GCM-ით
3. დაშიფრული ფაილის შენახვა UUID ფაილის სახელით
4. JSON მეტამონაცემების შენახვა (ორიგინალი ბილიკი, ჰეში, საფრთხის სახელი, დროის ანაბეჭდი)
5. აღდგენისას გაშიფვრა და მთლიანობის შემოწმება ჩაწერამდე

## აღმოფხვრის კონვეიერი

`--remediate` პარამეტრის გამოყენებისას:

```
საფრთხე აღმოჩენილია
  ├── 1. პროცესის შეწყვეტა     (SIGKILL Linux/macOS-ზე, TerminateProcess Windows-ზე)
  ├── 2. ფაილის კარანტინი       (AES-256-GCM დაშიფრული საცავი)
  └── 3. მუდმივობის გასუფთავება
        ├── Linux:   cron ამოცანები, systemd სერვისები, LD_PRELOAD
        ├── macOS:   LaunchAgents, plist ჩანაწერები, Keychain
        └── Windows: Run/RunOnce რეესტრი, დაგეგმილი ამოცანები, სერვისები
```

მოქმედებები კონფიგურირებადია `sd policy set` ბრძანებით.

## ხელმოწერების მონაცემთა ბაზა

### ჩაშენებული ხელმოწერები (`signatures-db/`)

მინიმალური ხელმოწერების ნაკრები, რომელიც ჩაშენებულია `sd` ბინარში `include_str!`-ის მეშვეობით:
- EICAR სატესტო ხელმოწერა
- ძირითადი YARA წესები (გამოსასყიდი პროგრამა, ტროიანი, backdoor და სხვ.)
- ცნობილი მავნე პროგრამების ჰეშები (WannaCry, Emotet, NotPetya)

### გარე ხელმოწერები ([prx-sd-signatures](https://github.com/openprx/prx-sd-signatures))

ყოვლისმომცველი, ხშირად განახლებადი საფრთხეების დაზვერვა:
- 38,800+ YARA წესი 9 წყაროდან
- ჰეშ ბლოკ-სიები abuse.ch არხებიდან
- IOC სიები: 585K+ მავნე IP-ები, დომენები, URL-ები

### საცავი

- **ჰეშები:** LMDB (heed crate) O(1) გასაღები-მნიშვნელობის ძებნისთვის
- **YARA წესები:** იტვირთება და კომპილირდება YARA-X-ით გაშვებისას
- **IOC სიები:** მეხსიერებაში HashSet სწრაფი IP/დომენის/URL შედარებისთვის

## მოდულების სისტემა

PRX-SD მხარს უჭერს WebAssembly მოდულებს Wasmtime-ის მეშვეობით:

- მოდულები არის `.wasm` ფაილები მანიფესტით (`plugin.json`)
- WASI მხარდაჭერა ფაილურ სისტემასა და გარემოზე წვდომისთვის
- მოდულების რეესტრი აღმოჩენისა და ჩატვირთვისთვის
- მასპინძელი ფუნქციები, რომლებიც მოდულებს აძლევს სკანირების შედეგებსა და კონფიგურაციაზე წვდომას

## განახლების სისტემა

`updater` crate და `update-server` უზრუნველყოფენ უსაფრთხო განახლების კონვეიერს:

1. კლიენტი ამოწმებს განახლების სერვერს ხელმოწერების ახალი ვერსიებისთვის
2. სერვერი პასუხობს ვერსიის ინფორმაციითა და ჩამოტვირთვის URL-ით
3. კლიენტი ჩამოტვირთავს zstd-კომპრესირებულ ხელმოწერების პაკეტს
4. პაკეტის ხელმოწერა ვერიფიცირდება Ed25519-ით (ed25519-dalek)
5. ხელმოწერები ამოიღება და იტვირთება LMDB-ში

## GUI აპლიკაცია

აგებულია Tauri 2 (Rust backend) + Vue 3 (TypeScript frontend) ტექნოლოგიებით:

- სისტემურ ტრეიში ინტეგრაცია სტატუსის ინდიკატორით
- საინფორმაციო პანელი საფრთხეების სტატისტიკით
- ფაილის სკანირება გადმოთრევით
- კარანტინის ბრაუზერი აღდგენისა და წაშლის ფუნქციებით
- რეალურ დროში მონიტორინგის მართვა
- პარამეტრები და კონფიგურაცია
- მრავალენოვანი მხარდაჭერა (10 ენა)
