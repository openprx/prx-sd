# არქიტექტურა

PRX-SD სტრუქტურირებულია როგორც Cargo workspace მოდულური crate-ებით, რომელთაგან თითოეული პასუხისმგებელია კონკრეტულ დომენზე.

## სამუშაო სივრცის განლაგება

```

├── crates/
│   ├── cli/           # "sd" ბინარი — ბრძანების ხაზის ინტერფეისი
│   ├── core/          # სკანირების ძრავის კოორდინაცია
│   ├── signatures/    # ჰეშების ბაზა (LMDB) + YARA-X წესების ძრავი
│   ├── parsers/       # ბინარული ფორმატების პარსერები
│   ├── heuristic/     # ევრისტიკული შეფასება + ML ინფერენცია
│   ├── realtime/      # ფაილური სისტემის მონიტორინგი + ქსელის ფილტრაცია
│   ├── quarantine/    # დაშიფრული კარანტინის საცავი

│   └── updater/       # ხელმოწერების განახლების კლიენტი
├── update-server/     # ხელმოწერების განაწილების სერვერი (Axum)

├── packaging/         # დისტრიბუციის შეფუთვა
├── tests/             # ინტეგრაციული ტესტები
├── tools/             # აწყობისა და სასარგებლო სკრიპტები

└── uninstall.sh       # დეინსტალაციის სკრიპტი
```

## აღმოჩენის კონვეიერი

სკანირების ძრავი (`core`) კოორდინაციას უწევს მრავალფენიანი აღმოჩენის კონვეიერს:

```
                    ┌──────────────┐
                    │  File Input  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │ Magic Number │  იდენტიფიკაცია: PE, ELF, MachO,
                    │  Detection   │  PDF, ZIP, Office, unknown
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
       ┌──────▼──────┐    │     ┌──────▼──────┐
       │    Hash      │    │     │   YARA-X    │
       │  Matching    │    │     │   Rules     │
       │  (LMDB)     │    │     │  (38K+)     │
       └──────┬──────┘    │     └──────┬──────┘
              │     ┌─────▼─────┐      │
              │     │ Heuristic │      │
              │     │ Analysis  │      │
              │     └─────┬─────┘      │
              │           │            │
              │    ┌──────▼──────┐     │
              │    │ ML Inference│     │
              │    │  (ONNX)    │     │
              │    └──────┬─────┘     │
              │           │            │
              │    ┌──────▼──────┐     │
              │    │ VirusTotal  │     │
              │    │ Cloud Query │     │
              │    └──────┬─────┘     │
              │           │            │
              └───────────┼────────────┘
                          │
                   ┌──────▼──────┐
                   │  Aggregate  │
                   │   Verdict   │
                   └─────────────┘
                   Clean / Suspicious / Malicious
```

### ფენების დეტალები

1. **ჰეშების შედარება** — O(1) ძიება LMDB მონაცემთა ბაზაში, რომელიც შეიცავს SHA-256 და MD5 ჰეშებს ClamAV-დან, abuse.ch-დან, VirusShare-დან და მორგებული შავი სიებიდან.

2. **YARA-X წესები** — შაბლონების შედარება YARA-X ძრავის გამოყენებით (Rust-ის მშობლიური YARA იმპლემენტაცია). წესები იტვირთება ჩაშენებული ნაგულისხმევი პარამეტრებიდან და გარე ხელმოწერების საცავიდან.

3. **ევრისტიკული ანალიზი** — ფაილის ტიპის მიხედვით ანალიზი:
   - **PE:** სექციის ენტროპია, საეჭვო API იმპორტები (CreateRemoteThread, VirtualAllocEx), შეფუთვის აღმოჩენა (UPX, Themida), დროის ნიშნულების ანომალიები
   - **ELF:** სექციის ენტროპია, LD_PRELOAD მიმართვები, cron/systemd მდგრადობა, SSH backdoor შაბლონები
   - **MachO:** სექციის ენტროპია, dylib ინექცია, LaunchAgent მდგრადობა, Keychain წვდომა

4. **ML ინფერენცია** (არჩევითი, feature flag `onnx`) — ONNX მოდელის შეფასება tract-ის მეშვეობით:
   - PE: 64-განზომილებიანი მახასიათებლების ვექტორი (იმპორტის ცხრილის ჰეშები, სექციის ენტროპია, API ხელმოწერები)
   - ELF: 48-განზომილებიანი მახასიათებლების ვექტორი (სექციის ენტროპია, სიმბოლოთა ცხრილი, დინამიკური ბიბლიოთეკები)

5. **VirusTotal ღრუბელი** — სარეზერვო ვარიანტი ფაილებისთვის, რომლებიც ლოკალურად არ შეესაბამება. იკითხება VirusTotal API (უფასო დონე: 500 მოთხოვნა/დღეში). შედეგები ინახება LMDB-ში.

### ქულების სისტემა

- ქულა >= 60: **მავნე (Malicious)**
- ქულა 30-59: **საეჭვო (Suspicious)**
- ქულა < 30: **სუფთა (Clean)**

საბოლოო ვერდიქტი არის უმაღლესი საფრთხის დონე ნებისმიერი აღმოჩენის ფენიდან.

## კარანტინის საცავი

ფაილები კარანტინში თავსდება AES-256-GCM ავთენტიფიცირებული დაშიფვრით:

1. შემთხვევითი 256-ბიტიანი გასაღებისა და 96-ბიტიანი nonce-ის გენერაცია
2. ფაილის შინაარსის დაშიფვრა AES-256-GCM-ით
3. დაშიფრული ფაილის შენახვა UUID ფაილის სახელით
4. JSON მეტამონაცემების შენახვა (ორიგინალი გზა, ჰეში, საფრთხის სახელი, დროის ნიშნული)
5. აღდგენა გაშიფრავს და ამოწმებს მთლიანობას ჩაწერამდე

## გამოსწორების კონვეიერი

როდესაც `--remediate` გამოიყენება:

```
Threat Detected
  ├── 1. Kill Process     (SIGKILL on Linux/macOS, TerminateProcess on Windows)
  ├── 2. Quarantine File  (AES-256-GCM encrypted vault)
  └── 3. Clean Persistence
        ├── Linux:   cron jobs, systemd services, LD_PRELOAD
        ├── macOS:   LaunchAgents, plist entries, Keychain
        └── Windows: Run/RunOnce registry, scheduled tasks, services
```

მოქმედებები კონფიგურირებადია `sd policy set`-ის მეშვეობით.

## ხელმოწერების მონაცემთა ბაზა

### ჩაშენებული ხელმოწერები (`signatures-db/`)

მინიმალური ხელმოწერების ნაკრები, რომელიც კომპილირებულია `sd` ბინარში `include_str!`-ის მეშვეობით:
- EICAR ტესტის ხელმოწერა
- ძირითადი YARA წესები (ransomware, trojan, backdoor და ა.შ.)
- ცნობილი მავნე პროგრამების ჰეშები (WannaCry, Emotet, NotPetya)

### გარე ხელმოწერები ([prx-sd-signatures](https://github.com/openprx/prx-sd-signatures))

ყოვლისმომცველი, ხშირად განახლებადი საფრთხეების დაზვერვა:
- 38,800+ YARA წესი 9 წყაროდან
- ჰეშების შავი სიები abuse.ch ფიდებიდან
- IOC სიები: 585K+ მავნე IP, დომენები, URL-ები

### შენახვა

- **ჰეშები:** LMDB (heed crate) O(1) გასაღები-მნიშვნელობის ძიებისთვის
- **YARA წესები:** იტვირთება და კომპილირდება YARA-X-ით გაშვებისას
- **IOC სიები:** მეხსიერებაში HashSet სწრაფი IP/დომენი/URL შედარებისთვის

## დანამატების სისტემა

PRX-SD მხარს უჭერს WebAssembly დანამატებს Wasmtime-ის მეშვეობით:

- დანამატები არის `.wasm` ფაილები მანიფესტით (`plugin.json`)
- WASI მხარდაჭერა ფაილურ სისტემასა და გარემოზე წვდომისთვის
- დანამატების რეესტრი აღმოჩენისა და ჩატვირთვისთვის
- ჰოსტის ფუნქციები ხელმისაწვდომია დანამატებისთვის სკანირების შედეგებისა და კონფიგურაციისთვის

## განახლების სისტემა

`updater` crate და `update-server` უზრუნველყოფენ უსაფრთხო განახლების კონვეიერს:

1. კლიენტი ამოწმებს განახლების სერვერს ხელმოწერების ახალი ვერსიებისთვის
2. სერვერი პასუხობს ვერსიის ინფორმაციითა და ჩამოტვირთვის URL-ით
3. კლიენტი ჩამოტვირთავს zstd-შეკუმშულ ხელმოწერების პაკეტს
4. პაკეტის ხელმოწერა მოწმდება Ed25519-ით (ed25519-dalek)
5. ხელმოწერები ამოიშლება და ჩაიტვირთება LMDB-ში

## GUI აპლიკაცია

აგებულია Tauri 2 (Rust backend) + Vue 3 (TypeScript frontend) ტექნოლოგიებით:

- სისტემური ტრეის ინტეგრაცია სტატუსის ინდიკატორით
- დაფა საფრთხეების სტატისტიკით
- ფაილების სკანირება გადაათრევ-ჩააგდებით
- კარანტინის ბრაუზერი აღდგენისა და წაშლის შესაძლებლობით
- რეალურ დროში მონიტორინგის მართვის ელემენტები
- პარამეტრები და კონფიგურაცია
- მრავალენოვანი მხარდაჭერა (10 ენა)
